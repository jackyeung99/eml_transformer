

General Ideas
- give context to what your are implementing 
- ask if there are better methods then the way you are doing 
- ask about trade offs for different decisions
- avoid confirmation bias by asking for approaches to problems without interjecting your own idea
- ask high-level architecture and design questions first before diving into implementation details or low-level coding prompts
- add constraints to prompts


### Sample Prompts 

- I want to incorporate natural language processing into energy demand forecasting. How should I structure my repository so it remains modular, scalable, and easy to extend with new data sources, preprocessing steps, embedding models, and forecasting pipelines over time?


- I want to design my codebase so it has clear entry points, modular components, and is easy to extend over time. What are the best practices for structuring a Python repository around pure functions while still supporting orchestration, configuration, pipelines, and scalable workflows? How should I think about layering abstractions and composing functionality without creating tightly coupled code?


- What are some different ways of organizing my data using a medallion architecture which will work down the line after changes 

- how do I write robust test cases that still work after changing code 


- How should I best go about having multiple modules for different sources that have similar behavior. How can I leverage Classes? 

- What are some approaches to uniquely identifying records 



- what are some text sources that may be beneficial to this goal
